# 使用 Xeno-Agent 模拟生产诊断系统

## 概述

Xeno-Agent 是一个通用的多 agent 协作模拟框架，可以用来模拟生产系统中的故障诊断流程。通过 offline simulation，我们可以：
- 验证诊断策略的有效性
- 训练和优化 agent 行为
- 模拟各种故障场景
- 提取最佳实践到生产系统

## 架构设计

### 生产系统诊断流程

```
故障现象收集 → 症状分析 → 根因诊断 → 方案生成 → 验证
     ↓              ↓            ↓           ↓          ↓
Symptom        Diagnosis     RootCause   Solution   Verification
Collector      Analyst       Finder       Generator    Agent
```

### Xeno-Agent 映射

使用 xeno-agent 的 GOTO (switch_mode) 和 GOSUB (new_task) 机制来模拟这个流程：

```yaml
主流程 (PM):
  1. 收集故障现象 (GOSUB → SymptomCollector)
  2. 分析症状并启动诊断 (GOSUB → DiagnosisAnalyst)
  3. 确认诊断结果并决定下一步
  4. 如果需要更多信息 (GOTO → SymptomCollector)
  5. 如果已找到根因 (GOSUB → SolutionGenerator)
  6. 验证解决方案 (GOSUB → VerificationAgent)
  7. 总结结果
```

### 4. 创建诊断场景脚本

创建 `examples/diagnosis_scenario.py`:

```python
"""
Example: Running a fault diagnosis simulation scenario.
"""

import os
from pathlib import Path
from xeno_agent.simulation import (
    register_builtin_skills,
    AgentRegistry,
    load_agent_from_yaml,
    create_crewai_llm,
    InteractionHandler,
    SimulationState,
    TaskFrame,
    XenoSimulationFlow,
)


def setup_diagnosis_agents(llm):
    """Load all diagnosis-related agents."""
    agents_dir = Path(__file__).parent.parent / "src" / "xeno_agent" / "simulation" / "agents"

    diagnosis_agents = [
        "symptom_collector_agent.yaml",
        "diagnosis_analyst_agent.yaml",
        "rootcause_finder_agent.yaml",
        "solution_generator_agent.yaml",
        "verification_agent.yaml"
    ]

    for agent_file in diagnosis_agents:
        agent_path = agents_dir / agent_file
        if agent_path.exists():
            try:
                load_agent_from_yaml(str(agent_path), llm=llm)
                print(f"✓ Loaded {agent_file}")
            except Exception as e:
                print(f"✗ Failed to load {agent_file}: {e}")


def run_diagnosis_scenario(
    incident_description: str,
    auto_approve: bool = True
):
    """
    Run a fault diagnosis simulation.

    Args:
        incident_description: Description of the incident/failure
        auto_approve: Whether to auto-approve tool executions
    """
    # Configure HITL
    if auto_approve:
        InteractionHandler.set_auto_approve(True)

    # Create LLM
    llm = create_crewai_llm()

    # Register built-in skills (includes diagnostic tools)
    register_builtin_skills()

    # Load diagnosis agents
    setup_diagnosis_agents(llm)

    # Create initial state with Symptom Collector as first step
    state = SimulationState(
        stack=[
            TaskFrame(
                mode_slug="symptom_collector",
                task_id="incident_001",
                trigger_message=f"""
故障事件: {incident_description}

请:
1. 收集所有相关的系统指标和日志
2. 识别关键症状和异常
3. 整理成结构化的症状报告
4. 将结果传递给诊断分析师进行分析
                """.strip(),
                caller_mode=None,
                is_isolated=False,
            )
        ],
        conversation_history=[
            {"role": "user", "content": incident_description}
        ],
        final_output=None,
        is_terminated=False,
        last_signal=None,
    )

    # Create and run flow
    flow = XenoSimulationFlow(state=state)

    print("\n" + "="*60)
    print("开始故障诊断模拟")
    print("="*60 + "\n")

    try:
        flow.kickoff()

        print("\n" + "="*60)
        print("诊断模拟完成")
        print("="*60)
        print(f"\n最终输出:\n{state.final_output}")
        return 0

    except KeyboardInterrupt:
        print("\n\n模拟被用户中断。")
        return 130
    except Exception as e:
        print(f"\n\n模拟失败: {e}")
        import traceback
        traceback.print_exc()
        return 1


if __name__ == "__main__":
    import sys

    # Example incident
    incident = """
从 10:15 开始，API 网关出现连接超时错误，错误率上升到 0.05。
用户反馈访问缓慢，部分请求失败。监控系统显示 CPU 使用率 85%。
数据库连接池接近上限。
    """

    # Run simulation
    sys.exit(run_diagnosis_scenario(incident, auto_approve=True))
```

### 5. 预期诊断流程

当运行诊断场景时，预期流程如下：

```
1. SymptomCollector (initial)
   - 调用 collect_metrics() → 收集 CPU, memory, errors, latency
   - 调用 query_logs() → 获取错误日志
   - 整理症状报告
   - new_task() → DiagnosisAnalyst

2. DiagnosisAnalyst
   - 接收症状报告
   - 调用 query_knowledge_base() → 查找历史案例
   - 分析症状，生成假设
   - 可能调用 deep_inspect() → 深入检查数据库配置
   - 识别根因：数据库连接池耗尽
   - attempt_completion() → 返回根因分析

3. SolutionGenerator (PM decides based on diagnosis)
   - 接收根因：连接池耗尽
   - 调用 query_knowledge_base() → 查找解决方案
   - 设计方案：增加池大小 + 添加泄漏检测
   - attempt_completion() → 返回解决方案

4. VerificationAgent
   - 设计测试用例
   - 模拟解决方案实施
   - 验证问题已解决
   - attempt_completion() → 完成验证

5. PM (final)
   - 总结整个诊断过程
   - 输出最终报告
   - attempt_completion() → simulation complete
```

## 总结

Xeno-Agent 提供了一个灵活的框架来模拟生产诊断系统：

### 核心优势

1. **模块化设计**：每个 agent 专注单一职责，易于扩展
2. **流程控制**：通过 switch_mode/new_task 灵活地模拟复杂诊断流程
3. **HITL 支持**：可以人工介入关键的诊断决策
4. **可测试性**：完全 offline，不影响生产环境
5. **经验积累**：从模拟中提取最佳实践

### 应用场景

- **离线测试新诊断策略** - 在不影响生产的情况下验证新方法
- **训练新 agent** - 通过模拟场景训练新的诊断角色
- **故障演练** - 模拟各种故障场景，准备应急预案
- **知识库验证** - 测试知识库的完整性和准确性
- **流程优化** - 分析诊断路径，减少不必要的步骤

### 与 Agent Gym 的协同

```
┌─────────────────┐     模式提取      ┌──────────────┐
│  Xeno-Agent    │ ─────────────────→ │  Agent Gym   │
│  (Offline)      │                  │ (Production)  │
└─────────────────┘                  └──────────────┘
       ↓                                      ↑
   模拟诊断                              应用新策略
       ↓                                      ↑
┌─────────────────┐                   ┌──────────────┐
│  Mock Data     │                 │  Real Data   │
│  Sources       │                 │  Sources     │
└─────────────────┘                   └──────────────┘
```

通过这个框架，可以持续改进生产系统的诊断能力，同时保证安全性和稳定性。
