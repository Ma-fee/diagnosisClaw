# 0005 委派引擎与可观测性：你如何证明系统“真的在协作”

## 你做了什么
你把多agent协作从“黑盒调用”做成了“可观测事件流”。

## 关键机制
1. 委派时创建 `child_session_id`，并记录 `parent_session_id` + `tool_call_id`。
2. 在子任务启动前发 `SpawnSessionStart`。
3. 子agent事件通过 `SubAgentEvent` 包装并回传。
4. 对 `attempt_completion` 专门捕获，确保最终结果稳定落地。
5. 配置层开启 observability（OTLP HTTP/protobuf，service_name=agentpool）。

## 为什么这很关键
1. 可定位：出现错误时能回溯到具体子任务和调用链。
2. 可度量：能统计委派深度、成功率、平均耗时。
3. 可治理：能发现“过度委派/循环委派/无效委派”。

## 面试官问题
### Q1：你怎么判断委派是否有效？
建议回答：
- 看三类指标：
  1) 子任务成功返回率；
  2) 尝试完成信号触发率；
  3) 父任务最终收敛率。
- 再结合链路耗时和重试率评估收益。

### Q2：为什么不用日志就好，还要事件模型？
建议回答：
- 日志是文本，结构弱、跨会话关联差。
- 事件模型有明确字段（会话、深度、source_type），天然适合追踪和可视化。

### Q3：如果子agent没调用 `attempt_completion` 怎么办？
建议回答：
- 代码已做兜底：捕获 `StreamCompleteEvent` 提取最终消息。
- `attempt_completion` 是强约定，不是唯一回收路径。

## Trade-off
1. 强事件建模提升可观测性，但实现复杂度更高。
2. 额外事件上报带来少量性能开销，但能显著降低排障成本。
