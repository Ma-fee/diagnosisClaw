# 0008 一面模拟问答清单（面试官视角）

## 1. 你这个项目一句话介绍？
建议答法：
- 我做的是一个面向设备故障诊断的多Agent系统，核心是“分工编排 + 技能注入 + 工具协议 + 人在回路”，把大模型从聊天能力提升为可执行、可追踪的诊断流程。

## 2. 你在里面最有技术含量的一点是什么？
建议答法：
- 我把 `new_task` 做成了“可携带技能的方法论委派”，并配合 `expected_output`、深度限制、事件追踪，保证子任务既能发挥又不失控。

## 3. Prompt 你做了哪些不是“写文案”的工程化设计？
建议答法：
1. capability/role 双层拆分。
2. prompt 与 schema 双重约束。
3. 按角色最小能力集装配，减少互相污染。

## 4. Skills 的价值是什么？
建议答法：
- skills 让“诊断方法”可复用、可组合、可按需注入，不再依赖单个超长 prompt。

## 5. Tools 里你最满意的设计？
建议答法：
- `question_for_user`：把复杂追问结构化为问卷 schema。
- `update_todo_list`：增量状态更新，让流程可见且可回放。

## 6. 你怎么保证不会无限委派？
建议答法：
- 代码里设了最大深度（5），超限即报错并终止链路，这是硬护栏。

## 7. 为什么要搞 observability？
建议答法：
- 多agent不是单链路调用，必须有父子会话和事件级追踪，不然线上问题无法定位。

## 8. 讲一个你做过的 trade-off。
建议答法：
- 我选择强约束工具协议而不是自由文本，虽然开发更重，但换来稳定性、可审计性和可自动化回归测试能力。

## 9. 如果让你再做一版 v6，会优先做什么？
建议答法：
1. 增加委派策略学习（何时委派的策略评估）。
2. 做技能版本治理与灰度。
3. 增加故障场景 benchmark 和回归集。

## 10. 你在团队中的角色是什么？
建议答法：
- 我不仅写了 prompt，还负责了编排配置、工具 schema、provider 代码和面向诊断场景的方法论落地，覆盖了“策略到实现”的全链路。
