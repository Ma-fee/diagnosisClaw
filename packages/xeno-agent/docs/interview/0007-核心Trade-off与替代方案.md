# 0007 核心Trade-off与替代方案：面试最容易追问的地方

## Trade-off 1：多Agent vs 单Agent
- 你的选择：多Agent。
- 代价：编排复杂度、调用延迟增加。
- 收益：职责清晰、上下文隔离、行为更可控。

可替代方案：
- 单Agent + 长prompt + tool路由。
- 适合小规模场景，但在复杂诊断中容易角色混叠。

## Trade-off 2：动态skills注入 vs 全量预加载
- 你的选择：按任务注入 `load_skills`。
- 代价：调用方要维护技能选择逻辑。
- 收益：降低 token 开销，减少指令冲突。

可替代方案：
- 固定技能包。
- 上手简单，但长期会出现“上下文噪声堆积”。

## Trade-off 3：强约束工具协议 vs 自由文本交互
- 你的选择：question/todo/new_task 等协议化工具。
- 代价：开发维护成本上升，灵活表达受限。
- 收益：可审计、可自动化、可测试。

可替代方案：
- 全文本对话 + 后处理抽取。
- 初期快，但稳定性和可追踪性不足。

## Trade-off 4：事件可观测性 vs 性能
- 你的选择：发射 spawn/subagent/complete 等事件。
- 代价：有额外上报和处理开销。
- 收益：链路可追踪，便于线上排障与优化。

可替代方案：
- 仅日志记录。
- 开发简单，但难以构建统一指标体系。

## 面试官追问模板
### Q1：如果让你把系统时延降低 30%，你会砍哪里？
建议回答：
1. 收缩不必要委派，优先本地直接回答简单问题。
2. 合并可串联的追问，减少交互轮数。
3. 对检索型子任务做并行与缓存。

### Q2：如果线上经常误判根因，你先改 prompt 还是改 tools？
建议回答：
- 先看错误归因：
  1) 信息不足导致误判 -> 改提问策略与必填字段；
  2) 推理偏差 -> 改诊断技能与阶段约束；
  3) 数据源问题 -> 改检索与引用校验。
- 原则是“先改最小闭环中的硬约束，再调软约束”。

### Q3：你这套系统最难被替代的工程价值是什么？
建议回答：
- 不是“会调用 LLM”，而是把诊断知识、交互协议、编排治理和可观测性打通成一个可持续演进的系统。
