# 用户交互机制

OpenCode 提供了两种截然不同的用户交互机制：**主动提问 (Active Questioning)** 和 **被动确认 (Passive Confirmation)**。

## 1. 主动提问 (Question Tool)

Agent 可以通过 `question` 工具主动向用户发起结构化提问。

**文件位置**: `packages/opencode/src/tool/question.ts`

### 工具定义

```typescript
// Tool Schema
{
  question: "完整的问题内容...",
  header: "短标题(12字符内)",
  options: [
    { label: "选项A", description: "解释..." },
    { label: "选项B", description: "解释..." }
  ],
  multiple: false, // 是否多选
  custom: true     // 是否允许用户输入自定义答案
}
```

### 权限控制

该工具受 `permission.question` 规则控制：

- **Default**: `deny` (防止 Agent 随意打扰用户)
- **Build Agent**: `allow`
- **Plan Agent**: `allow`

### 使用场景

- 澄清模糊的需求 ("Which framework should I use?")
- 选择执行路径 ("Found 2 implementations. Which one to refactor?")
- 确认重大变更 ("This will delete 50 files. Proceed?")

### 实现机制

1. Agent 调用 `question` 工具
2. 系统发布 `question.asked` 事件
3. 客户端 (CLI/TUI) 捕获事件并展示交互界面
4. Agent 挂起，等待 `question.replied` 事件
5. 用户选择或输入后，结果作为工具输出返回给 Agent

## 2. 被动确认 (Permission System)

当 Agent 尝试执行敏感操作时，权限系统会自动拦截并请求用户确认。

**文件位置**: `packages/opencode/src/permission/next.ts`

### 触发机制

在 `Agent.permission` 配置中设置为 `ask` 的规则：

```typescript
{
  "permission": {
    "bash": { "rm *": "ask" },    // 删除文件需确认
    "read": { "*.env": "ask" },   // 读取密钥需确认
    "doom_loop": "ask"            // 死循环检测
  }
}
```

### 用户选项

当触发 `ask` 时，用户面临三个选择：

1. **Allow Once**: 仅允许本次调用
2. **Always Allow**: 允许本次及后续所有符合模式的调用（Session 级缓存）
3. **Deny**: 拒绝本次调用（工具抛出 `PermissionDeniedError`）

### 与 Question 工具的区别

| 特性       | Question Tool         | Permission Ask                   |
| ---------- | --------------------- | -------------------------------- |
| **发起方** | Agent 主动发起        | 系统被动触发                     |
| **目的**   | 获取信息/决策         | 安全确认                         |
| **内容**   | 自由文本 + 选项       | "Allow tool [name] with [args]?" |
| **结果**   | 文本/选项             | Allow / Deny                     |
| **配置**   | `permission.question` | 具体的工具规则                   |

## 3. 特殊交互场景

### Plan Mode 切换

OpenCode 使用专门的工具 `plan_enter` 和 `plan_exit` 来管理模式切换的确认。

- **plan_enter**: 询问 "Enter plan mode?"
- **plan_exit**: 询问 "Exit plan and apply changes?"

这两个工具内部实际上是封装了 `Question.ask()`，但拥有独立的权限控制 (`permission.plan_enter/exit`)。

### CLI 直接输入

在 CLI 层面 (`src/cli/ui.ts`)，还提供了底层的 `UI.input()` 函数，用于在启动阶段或非 Agent 循环中获取用户输入。但这不是 Agent 可调用的工具。

## 4. ACP Session 中的 Question Tool

### ACP 协议适配

**核心机制**：Question Tool 在 ACP Session 中没有特殊的适配逻辑，被当作普通工具通过标准 ACP 协议传输。

**事件流转**：
```
Agent 调用 question
  ↓
Question.ask() 发布 `question.asked` 事件
  ↓
ACP Agent 订阅 SDK 事件
  ↓
通过 sessionUpdate 机制传输:
  - "pending" → sessionUpdate: "tool_call"
  - "running" → sessionUpdate: "tool_call_update" (in_progress)
  - "completed" → sessionUpdate: "tool_call_update" (completed)
  - "error" → sessionUpdate: "tool_call_update" (failed)
  ↓
ACP 客户端接收状态更新
```

**关键文件**：
- `packages/opencode/src/tool/registry.ts:97` - 工具注册（条件控制）
- `packages/opencode/src/acp/agent.ts` - ACP 事件处理机制
- `packages/opencode/src/acp/session.ts` - ACP Session 状态管理

### 协议适配细节

在 `packages/opencode/src/acp/agent.ts` 中，Question Tool 通过工具类型映射函数：

```typescript
// Question Tool 映射为 "other" 类型
function toToolKind() {
  // Question Tool 作为普通工具处理
  return "other"
}
```

**限制说明**：
- ACP 客户端（如 Zed）可能没有 UI 支持来显示交互式问题
- 即使服务端发送了 `question.asked` 事件，客户端也可能无法渲染问题界面
- 这导致 Question Tool 在 ACP 环境中某种程度上"不可用"

### 独立开关控制

Question Tool 可以通过 `OPENCODE_CLIENT` 环境变量独立控制：

**注册逻辑** (`packages/opencode/src/tool/registry.ts:97`)：
```typescript
...(["app", "cli", "desktop"].includes(Flag.OPENCODE_CLIENT) ? [QuestionTool] : []),
```

**环境变量定义** (`packages/opencode/src/flag.ts:22`)：
```typescript
export const OPENCODE_CLIENT = process.env["OPENCODE_CLIENT"] ?? "cli"
```

**当前状态**：
- ACP Session 默认启用 Question Tool（继承 `OPENCODE_CLIENT="cli"`）
- 由于 `OPENCODE_CLIENT` 默认为 `"cli"`，ACP 和普通 CLI 共享相同的工具集

**禁用方法**：
```bash
# 方法 1: 环境变量方式
OPENCODE_CLIENT=acp opencode acp

# 方法 2: 修改 ACP 命令（packages/opencode/src/cli/cmd/acp.ts）
process.env["OPENCODE_CLIENT"] = "acp"
```

**推荐做法**：
考虑到 ACP 客户端的 UI 限制，建议在 ACP 模式下**默认禁用** Question Tool：
1. 修改 ACP 命令启动逻辑，设置 `OPENCODE_CLIENT="acp"`
2. 保持 CLI/App/Desktop 的 Question Tool 功能不受影响
3. 这样可以避免 Agent 在 ACP 环境中调用无效的交互工具

### 与普通 Session 的差异

| 特性 | 普通 Session | ACP Session |
|------|-------------|------------|
| **注册方式** | 通过 ToolRegistry 动态加载 | 同左 |
| **事件发布** | `question.asked` 事件 | 同左 |
| **传输协议** | SDK 自定义协议 | ACP 标准协议 (sessionUpdate) |
| **UI 支持** | CLI/App/Desktop 完整支持 | 取决于客户端实现 |
| **当前状态** | 默认启用 | 默认启用，但可能无 UI |
| **独立开关** | 受 `OPENCODE_CLIENT` 控制 | 同左 |

## 总结

OpenCode 的交互设计遵循 **"打扰最小化，安全最大化"** 的原则：

- 默认情况下 Agent 不能随意提问 (`question: deny`)
- 敏感操作必须经过确认 (`ask` rules)
- 提供了结构化的提问能力，优于自由文本对话
- **ACP 环境需要特别处理**：Session 支持但客户端 UI 可能缺失
