# 工具系统设计

## 核心架构

OpenCode 的工具系统采用**声明式定义**和**运行时验证**的架构。所有工具通过 `Tool.define()` 创建，并强类型绑定到 Zod Schema。

## 工具定义结构

### Tool Interface

```typescript
namespace Tool {
  // 定义工具的上下文
  type Context = {
    sessionID: string;
    messageID: string;
    agent: string;
    abort: AbortSignal;
    callID?: string;
    metadata(input: { title?: string; metadata?: M }): void;
    ask(input: PermissionRequest): Promise<void>;
  };

  // 定义工具的参数和元数据
  interface Info<Params extends ZodType, Meta extends Metadata> {
    id: string;
    init: (ctx?: InitContext) => Promise<{
      description: string;
      parameters: Params;
      execute(
        args: Params,
        ctx: Context,
      ): Promise<{
        title: string;
        metadata: Meta;
        output: string;
        attachments?: FilePart[];
      }>;
      formatValidationError?(error: ZodError): string;
    }>;
  }
}
```

### 定义工具示例

```typescript
export const ReadTool = Tool.define("read", {
  description: "Reads a file from the local filesystem...",
  parameters: z.object({
    filePath: z.string().describe("The absolute path to the file"),
    limit: z.number().optional().default(2000),
    offset: z.number().optional().default(0),
  }),
  execute: async ({ filePath, limit, offset }, ctx) => {
    // 1. 验证和读取
    const content = await fs.readFile(filePath, "utf-8");

    // 2. 返回结果
    return {
      title: `Read ${filePath}`,
      metadata: { size: content.length },
      output: content,
    };
  },
});
```

## 核心特性

### 1. 自动截断 (Auto-Truncation)

为了防止 LLM 上下文溢出，工具系统内置了自动截断机制：

```typescript
// Tool.define 内部包装器
const truncated = await Truncate.output(result.output, {}, initCtx?.agent);
return {
  ...result,
  output: truncated.content,
  metadata: {
    ...result.metadata,
    truncated: truncated.truncated,
    ...(truncated.truncated && { outputPath: truncated.outputPath }),
  },
};
```

**策略**:

- 如果输出过长，会被自动截断
- 完整内容会被写入临时文件
- 返回给 LLM 的是截断后的内容 + 完整内容的路径
- 工具可以自行处理截断（通过返回 `metadata.truncated`）

### 2. 模式验证 (Schema Validation)

- 使用 Zod 进行严格的参数验证
- 自定义 `formatValidationError` 提供更友好的错误信息
- 验证失败直接抛出错误，LLM 会收到错误提示并尝试修复

### 3. 上下文注入

每个工具执行时都会注入 `Context` 对象：

- `sessionID` / `messageID` - 用于关联日志和事件
- `abort` - 支持取消长时间运行的操作
- `ask()` - 用于动态申请权限

### 4. 附件支持 (Attachments)

工具不仅返回文本，还可以返回附件：

```typescript
attachments: [
  {
    type: "file",
    mime: "image/png",
    url: "...",
  },
];
```

这对于多模态支持（如读取图片）至关重要。

## 内置工具集

OpenCode 提供了 25+ 个内置工具，涵盖以下类别：

### 基础操作

- `read`: 读取文件
- `write`: 写入文件 (需读取后)
- `edit`: 编辑文件 (需读取后)
- `bash`: 执行 Shell 命令 (支持 timeout, workdir)
- `glob`: 查找文件
- `grep`: 搜索内容

### 高级功能

- `ls`: 列出目录
- `mv`: 移动文件
- `rm`: 删除文件 (需确认)
- `mkdir`: 创建目录

### 智能体协作

- `task`: 调用子 Agent 执行任务
- `call_omo_agent`: 另一个子 Agent 调用接口

### Web 能力

- `webfetch`: 获取网页内容 (支持 markdown/text/html)
- `websearch`: 搜索网络

### LSP 集成

- `lsp_hover`: 获取悬停信息
- `lsp_goto_definition`: 跳转定义
- `lsp_find_references`: 查找引用
- `lsp_document_symbols`: 文档符号
- `lsp_workspace_symbols`: 工作区符号
- `lsp_diagnostics`: 获取诊断信息
- `lsp_rename`: 重命名符号

### 其他

- `todowrite` / `todoread`: 管理任务列表
- `mcp-router_*`: MCP 协议路由工具

## Task 工具详解

Task 工具是实现多 Agent 协作的核心：

```typescript
// packages/opencode/src/tool/task.ts (伪代码)
execute: async ({ prompt, agent, ...args }, ctx) => {
  // 1. 创建子 Session
  const session = await Session.create({
    agent: agent || "general",
    parentSessionID: ctx.sessionID,
    // ...
  });

  // 2. 禁用某些工具 (防止递归问题)
  session.disableTools(["todowrite", "todoread"]);

  // 3. 执行 Prompt
  await SessionPrompt.prompt(session, prompt);

  // 4. 等待完成并返回结果
  return {
    output: "Task completed...",
    metadata: { sessionID: session.id },
  };
};
```

**特点**:

- 创建独立的子会话
- 支持指定 Agent 类型
- 支持传递上下文
- 结果包含 `session_id` 方便后续引用

## 工具注册机制

工具注册在 `ToolRegistry` 中，支持动态加载：

1. **静态注册**: 内置工具在启动时注册
2. **插件注册**: 插件可以通过 `hooks.tool.register` 添加新工具
3. **MCP 桥接**: 自动将 MCP 服务器暴露的工具桥接到 OpenCode 工具系统

## 最佳实践

1. **原子性**: 工具应做一件具体的事
2. **幂等性**: 尽可能设计为幂等的，方便重试
3. **安全性**: 所有危险操作必须通过权限系统检查
4. **可观察性**: 使用 `ctx.metadata()` 报告执行状态
