# 权限系统设计

## 核心理念

OpenCode 的权限系统基于**规则匹配 (Rule-Based Matching)** 和 **用户交互 (User Interaction)**。

设计目标：

- **安全优先**: 默认拒绝未知操作
- **细粒度**: 支持按工具、按参数、按路径控制
- **可配置**: 用户可自定义规则
- **交互式**: 支持 `ask` 模式，动态请求用户批准

## 权限模型

### Permission Rule

```typescript
interface Rule {
  permission: string; // 权限类型 (通常是工具名或资源类)
  action: "allow" | "deny" | "ask";
  pattern?: string | string[]; // 匹配模式 (glob)
}
```

### Permission Info (请求对象)

```typescript
interface Info {
  id: string;
  type: string; // 权限类型
  pattern?: string | string[]; // 请求的资源模式
  sessionID: string;
  messageID: string;
  callID?: string; // 关联的工具调用ID
  message: string; // 给用户的提示信息
  metadata: Record<string, any>;
}
```

## 运作流程

```
Tool.execute()
     ↓
ctx.ask({ type: "read", pattern: "/etc/passwd" })
     ↓
Permission.ask()
     ↓
1. 检查已批准缓存 (Approved Cache)
   → 如果命中，直接通过
     ↓
2. 触发插件 Hook (permission.ask)
   → 插件可以直接 allow/deny
     ↓
3. 创建 Pending Request
   → 状态变为 "pending"
   → 发布 BusEvent (permission.updated)
     ↓
4. 等待用户响应 (UI 层展示请求)
   → 用户点击 Allow / Deny / Always
     ↓
5. Permission.respond()
   → 更新状态
   → 如果是 Always，写入 Approved Cache
     ↓
6. Promise Resolve/Reject
   → Tool 继续执行 或 抛出 RejectedError
```

## 默认规则集

系统内置了一套安全的默认规则：

```typescript
const defaults = {
  // 默认允许所有工具 (但在具体工具内部会有更细的检查)
  "*": "allow",

  // 死循环检测 - 防止无限循环烧钱
  doom_loop: "ask",

  // 外部目录访问 - 防止访问工作区以外的文件
  external_directory: {
    "*": "ask",
    [Truncate.DIR]: "allow", // 允许访问临时截断目录
    [Truncate.GLOB]: "allow",
  },

  // 敏感文件读取
  read: {
    "*": "allow",
    "*.env": "ask", // 环境变量文件需确认
    "*.env.*": "ask",
    "*.env.example": "allow", // 示例文件允许
  },

  // 交互式询问
  question: "deny", // 默认不允许主动骚扰用户

  // Plan 模式控制
  plan_enter: "deny",
  plan_exit: "deny",
};
```

## 规则匹配逻辑

使用 `remeda` 库进行深度合并，优先级从低到高：

1. **System Defaults**: 硬编码的安全底线
2. **Agent Rules**: Agent 特有的规则 (如 explore 禁止写操作)
3. **User Config**: `~/.config/opencode/config.json` 中的配置
4. **Project Config**: 项目根目录的 `opencode.jsonc`

## 用户响应类型

当触发 `ask` 时，用户有三种选择：

- **Once**: 仅允许本次操作
- **Always**: 允许本次及后续相同模式的操作 (Session 级别缓存)
- **Reject**: 拒绝本次操作

### Always 的实现

```typescript
if (response === "always") {
  // 1. 记录到 approved cache
  approved[sessionID][pattern] = true

  // 2. 自动批准当前 pending 队列中符合模式的其他请求
  for (const item of pending) {
    if (covered(item.pattern, approved)) {
      respond({ ..., response: "always" })
    }
  }
}
```

## 特殊权限类型

除了工具名，还有一些特殊的权限类型：

- `doom_loop`: 检测到代理陷入重复循环
- `external_directory`: 访问工作区根目录之外的文件
- `question`: 代理主动向用户提问
- `plan_enter` / `plan_exit`: 切换 Agent 模式
- `run_command`: 执行 bash 命令 (通常由 bash 工具触发)

## 插件集成

插件可以拦截权限请求：

```typescript
// 插件示例
export default {
  hooks: {
    permission: {
      ask: async (info, ctx) => {
        // 自动允许读取 node_modules
        if (info.type === "read" && info.pattern?.includes("node_modules")) {
          return { status: "allow" };
        }
        return { status: "ask" }; // 继续默认流程
      },
    },
  },
};
```

## 实现细节

**文件位置**: `packages/opencode/src/permission/index.ts`

**关键类**:

- `Permission.ask()` - 发起请求
- `Permission.respond()` - 响应请求
- `Permission.list()` - 列出待处理请求

**状态管理**:
使用 `Instance.state` 维护：

- `pending`: 当前等待用户确认的请求
- `approved`: 当前会话已永久批准的模式

## 安全建议

1. **最小权限原则**: Agent 应只拥有完成任务所需的最小权限
2. **显式优于隐式**: 危险操作应显式列出
3. **环境隔离**: 敏感文件 (.env, id_rsa) 默认应设为 `ask` 或 `deny`

这是构建安全可靠的 Agent 系统的基石。
